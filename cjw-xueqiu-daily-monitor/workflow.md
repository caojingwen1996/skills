# Xueqiu Daily Monitor Workflow

## 1. 目标

这套流程用于支持一个当前明确、可执行的工作模式：

- 监控对象是多个雪球博主
- 博主主页链接由人工维护
- 任务开始时间由人工决定
- 每次手动启动只执行一轮抓取
- 同一天可以重复手动启动，但必须复用当天状态，只补抓新增内容
- 当天抓取结束后，再单独生成 Markdown 汇总

当前目标不是自动调度，也不是自动执行 `spec` 规则，而是先把“抓取完整、同日不重、汇总可读、过程可追溯”这四件事做好。

## 2. 范围与边界

### 2.1 当前包含

- 多博主输入管理
- 单博主、单日期抓取
- 同日续跑
- 当天去重
- 原始帖子落盘
- 中间分析文件保留
- 单博主日报
- 全博主总汇总

### 2.2 当前不包含

- 自动定时调度
- 小时级循环扫描
- 跨日连续运行
- 自动修改 `spec`
- 自动执行交易决策

## 3. 角色分工

### 3.1 `EXTEND.md`

职责：

- 维护博主主页链接
- 标记博主是否启用
- 作为当天任务的输入来源

约束：

- 这里只放人工配置
- 不记录运行状态
- 不记录汇总结果

### 3.2 `scripts/content_task.py`

职责：

- 面向单个博主、单个日期执行抓取
- 进入主页、识别目标日期帖子、打开详情、提取正文、保存原始文件

约束：

- 只负责内容抓取
- 不负责多博主调度
- 不负责最终汇总

### 3.3 `scripts/task_store.py`

职责：

- 维护“博主 + 日期”的状态文件
- 记录已处理内容
- 作为同日去重的唯一事实来源

约束：

- 只管理状态，不做内容理解

### 3.4 `scripts/daily_summary.py`

职责：

- 读取当天全部原始帖子
- 生成中间分析文件
- 输出单博主日报和全博主总汇总

约束：

- 不重新抓网页
- 不修改当天抓取状态

## 4. 输入与启动规则

### 4.1 输入来源

当天任务的输入由两部分组成：

1. `EXTEND.md` 中启用的博主主页链接
2. 本次任务的目标日期

目标日期规则：

- 默认是当天
- 也允许人工指定历史某一天

### 4.2 手动启动

当前流程是手动启动，不自动调度。

这意味着：

- 什么时候开始抓，由操作者决定
- 同一天是否再补抓一次，也由操作者决定
- 什么时候生成汇总，也由操作者决定

## 5. 日常启动 SOP

这是每天第一次启动任务时应遵循的标准步骤。

### 步骤 1：展示 `EXTEND.md` 当前配置并确认是否需要补充

启动任务前，必须先读取 `EXTEND.md`，并把当前人工配置整理后展示给操作者确认。

至少应列出：

- 当前启用的博主
- 当前禁用的博主
- 每个博主的主页链接
- 每个博主的备注（如果有）
- 当前手动启动规则
- 当前目标日期规则
- 当前输出目录约定

展示后必须明确询问操作者：

- 是否需要补充或修正 `EXTEND.md` 配置

处理规则：

- 如果操作者要求补充或修正，先更新 `EXTEND.md`
- 在 `EXTEND.md` 补充完成前，不进入后续抓取准备阶段
- 只有在操作者明确确认“无需补充”或补充完成后，才能继续后续步骤

完成标准：

- 可以明确回答“当前配置里启用了谁、禁用了谁、链接是否齐全、是否还需要补充”

### 步骤 2：检查输入

启动前必须确认：

- `EXTEND.md` 中需要跟踪的博主都已启用
- 每个博主都有有效主页链接
- 目标日期正确
- Chrome 持久化 profile 可用
- 雪球登录状态可用，或者已准备好人工登录

如果出现以下情况，则本轮任务不能继续启动：

- `EXTEND.md` 中没有任何启用的账户链接
- 已启用账户缺少主页链接

处理规则：

- 直接提示操作者先到 `EXTEND.md` 中补充或修正账户链接
- 在链接配置完成前，不进入抓取阶段

完成标准：

- 可以明确回答“今天抓谁、抓哪一天”

### 步骤 3：检查当天状态

对每个启用博主，先检查当天是否已存在：

- 状态文件
- 日志文件
- 原始结果目录

判断规则：

- 如果不存在，说明今天还没开始，进入“新任务模式”
- 如果已存在，说明今天跑过，进入“同日续跑模式”

完成标准：

- 每个博主都能明确归类为“新任务”或“续跑”

### 步骤 4：执行单轮抓取

对每个启用博主执行一轮抓取，顺序如下：

1. 打开博主主页
2. 扫描可见帖子
3. 识别属于目标日期的候选内容
4. 与当天已处理记录比对
5. 只打开新增帖子的详情页
6. 提取正文和基础字段
7. 保存原始帖子文件
8. 更新状态、日志、计数

完成标准：

- 当轮扫描到的目标日期新增帖子都已处理
- 已处理内容不会重复保存

### 步骤 5：结束本轮

一轮抓取结束后，不自动进入下一轮。

当前策略是：

- 一次手动启动只做一轮
- 是否再次启动，由操作者另行决定

完成标准：

- 当前轮的新增内容都已入库
- 状态文件已更新，便于当天后续续跑

## 6. 同日续跑 SOP

同一天再次启动时，不是“重跑一遍”，而是“在当天已有基础上补抓新增”。

### 续跑判断

只要满足以下条件，就视为同日续跑：

- 博主相同
- 目标日期相同
- 当天状态文件已存在

### 续跑步骤

1. 读取当天状态文件
2. 加载当天已处理内容集合
3. 再次扫描博主主页
4. 只保留当天尚未处理的新帖子
5. 保存新增内容
6. 追加日志和状态记录

### 续跑约束

- 不能覆盖已有原始文件
- 不能重复写入同一条帖子
- 不能把当天续跑当成新任务清空重来

完成标准：

- 第二次及之后的同日启动，只会增加新内容，不会制造重复内容

## 7. 去重规则

去重是这套流程最重要的保护机制。

### 去重范围

去重范围按以下维度隔离：

- 博主
- 日期
- 内容项

也就是说：

- 不同博主之间互不影响
- 不同日期之间互不影响
- 同一博主同一天内，同一条帖子只能处理一次

### 去重依据

当天状态文件中的已处理集合，是唯一事实来源。

这意味着：

- 是否重复，不看操作者记忆
- 不看文件名猜测
- 只看状态文件中的已处理记录

## 8. 日终汇总 SOP

抓取和汇总必须分开。

只有当操作者认为“当天抓取已足够完整”时，才进入汇总阶段。

### 步骤 1：冻结当天抓取范围

汇总前先确认：

- 今天是否还需要再补抓
- 当前原始帖子目录是否已经稳定

一旦开始做最终汇总，默认以当前原始数据为准。

### 步骤 2：读取当天全部原始帖子

汇总输入来自：

- `output/` 下各博主当天的原始帖子目录

汇总脚本不重新抓网页，只消费已经保存好的原始文件。

### 步骤 3：生成中间文件

所有中间文件保留在：

```text
output/processing/{yyyymmdd}/
```

当前中间层包括：

- `raw_index/`
- `normalized/`
- `extracted/`
- `post_analysis/`
- `author_daily/`
- `market_daily/`

这些文件的作用是：

- 保持可追溯
- 方便复核
- 为以后接入 `spec` 做准备

### 步骤 4：生成最终汇总

最终汇总只写入：

```text
output/summaries/{yyyymmdd}/
```

当前 final 文件包括：

- 每个博主一份 Markdown 日报
- 一份 `daily_summary.md` 全博主总汇总

完成标准：

- `summaries/{yyyymmdd}` 里只存在最终可读结果
- `processing/{yyyymmdd}` 里保留完整中间产物

## 9. 汇总输出结构

单博主日报和全博主总汇总都使用 Markdown。

### 9.1 核心内容

回答：

- 今天这个博主或这批博主到底说了什么

关注点：

- 主要观点
- 重点标的
- 明确交易表态

### 9.2 背景语境

回答：

- 为什么今天会说这些内容

关注点：

- 财报、公告、价格波动、做空、市场环境、上下文关系

### 9.3 Spec相关

只在相关时输出，分为两层：

- `明确相关`
  - 已经能明确对应现有规则
- `候选相关`
  - 暂时还不能确认，但值得后续人工复核

当前阶段允许 `明确相关` 为空。

## 10. 输出目录规则

所有输出统一放在 `output/` 下。

### 10.1 原始抓取层

每个博主每天一组：

```text
output/
├── 某博主_20260325.log
├── 某博主_20260325.state.json
└── 某博主_20260325/
    ├── 112100_xxx.txt
    ├── 150800_xxx.txt
    └── ...
```

### 10.2 中间处理层

```text
output/processing/{yyyymmdd}/
```

这里只放中间分析文件，不放 final。

### 10.3 最终结果层

```text
output/summaries/{yyyymmdd}/
```

这里只放最终 Markdown，不放中间文件。

## 11. 异常处理 SOP

以下情况都视为需要人工介入的异常：

- 登录失效
- 页面异常
- 页面结构变化
- 内容为空
- 保存失败
- 风控提示出现

### 处理原则

1. 先记录异常
- 异常必须进入日志和状态，不允许静默吞掉

2. 保留已完成成果
- 已经抓到的内容不能因为后续异常被破坏

3. 当前博主可中断，整天流程不必整体报废
- 单条失败不影响其他条继续
- 单个博主异常不应破坏其他博主的结果

4. 允许人工处理后继续同日续跑
- 修复登录或页面问题后，可再次手动启动当天任务

完成标准：

- 异常被看见、被记录、可恢复

## 12. 当天任务完成标准

以下条件满足时，可以认为“当天任务完成”：

- 需要抓取的博主都至少完成过一轮
- 同日补抓已经结束
- 当天原始帖子已稳定
- 单博主日报已生成
- 全博主总汇总已生成
- 关键异常已被记录

这时当天流程可以结束，等待下一次人工启动新的日期任务。

## 13. 核心原则

整个流程始终遵守以下原则：

1. 手动优先
- 当前阶段优先保证可控，而不是追求自动化

2. 同日不重
- 同一天重复启动，只补抓新增

3. 抓取与汇总分离
- 先积累原始数据，再做汇总

4. 结果可追溯
- 原始文件、中间文件、最终文件都要保留

5. `summaries` 只放 final
- 最终目录必须干净、可直接阅读
