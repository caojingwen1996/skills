# Obsidian KB

## 作用

把当前仓库视为一个长期维护的 Obsidian 知识库，而不是临时记事本。

这个 skill 负责：

- 将新的资料导入知识库
- 先判断资料是否属于当前知识库的 domain，以及是否应进入正式知识层
- 优先更新已有知识页，避免重复创建页面
- 基于已整理的知识页回答问题
- 审计并改进知识库健康状况
- 在知识库缺少核心文件时完成初始化

这个 skill **不负责** 定义知识库的具体结构细则。  
所有结构规则、页面语义、命名规则、frontmatter 规则、标签体系、建页阈值、链接约束，都应以 `SCHEMA.md` 为准。

## 何时使用

当用户有以下需求时，使用这个 skill：

- 将新的文章、PDF、转录稿、网页、笔记或其他资料导入知识库
- 更新已有的 `people/`、`concepts/`、`topics/`、`views/` 页面，而不是重复建页
- 提出应该基于知识库回答的问题，并且结果可能需要回写到 `queries/` 或 `comparisons/`
- 审计、整理、清理、重构知识库
- 首次初始化知识库目录与基础文件

以下情况不要使用这个 skill：

- 没有长期价值的临时草稿
- 不应进入知识库的一次性查询结果
- 无法被链接进知识网络的孤立笔记

## 触发语句

以下请求可视为直接触发：

- `整理知识库`
- `导入资料到 Obsidian`
- `更新人物页`
- `更新主题页`
- `沉淀观点`
- `知识库体检`
- `修复 wikilinks`
- `补全 frontmatter`
- `重建 index`
- `清理孤儿页面`

## 规则来源

始终将 `SCHEMA.md` 视为知识库规则的唯一准绳。

如果 `SKILL.md` 与 `SCHEMA.md` 出现不一致：

1. 以 `SCHEMA.md` 为准
2. 尽量保持向后兼容
3. 不要自行发明新规则，除非用户明确要求修改 schema
4. 如果 schema 缺口导致无法执行，只做最小且安全的假设，并明确说明

## 必做启动流程

在创建或修改任何知识页之前，必须先完成以下步骤：

1. 读取 `SCHEMA.md`
2. 读取 `index.md`
3. 读取 `log.md` 最近 `20-30` 条记录
4. 判断资料是否属于 `SCHEMA.md` 定义的 domain，或是否明确服务于 schema 允许的长期主线
5. 只有在通过 domain 判断后，才搜索是否已经存在相关页面
6. 判断应该更新已有页面还是创建新页面

在完成这些步骤之前，不允许直接创建新页面。

## 工作原则

- 优先更新已有页面，不轻易新建页面
- 先判断是否应进入正式知识层，再判断 page type
- 执行时遵循 `SCHEMA.md`，不要临时重解释知识库结构
- 以提升理解、提高复用性和保持一致性为目标
- 只要已有页面可以增强，就不要重复建页
- 保留来源可追溯性
- 遇到冲突信息时保留差异，不强行抹平
- 保持表达简洁、结构化、可维护
- 将原始资料、整理后知识页、索引页、日志页视为不同层，各自承担不同职责
- 不属于当前 domain 的资料，直接丢弃，不进入 `raw/` 或正式知识页
- 不要在未说明的情况下悄悄改变知识库约定
- 不要把这个知识库当成一次性笔记堆放区

## 核心任务类型

这个 skill 主要执行三类任务：

### 1. Ingest
把新的资料纳入知识库，并连接到已有知识网络中。

### 2. Query
基于已整理的知识页回答问题，并在需要时把高价值答案回写到知识库。

### 3. Lint
检查知识库健康状况，识别结构、链接、元数据或维护层面的问题。

## Ingest 工作流

当收到新的文章、网页、访谈、PDF、笔记或其他资料时：

1. 读取 `SCHEMA.md`、`index.md` 和最近的 `log.md`
2. 先判断资料是否属于 `SCHEMA.md` 定义的 domain，或是否明确服务于 schema 允许的长期主线
3. 若不属于当前 domain，或虽已归档候选但不应进入正式知识层，则直接丢弃，不写入 `raw/`，并在任务结尾明确说明“未入库”
4. 只有在通过 domain 判断后，才搜索是否已有相关页面
5. 将原始资料归档到正确的 `raw/` 子目录
6. 按 schema 要求提取元信息与候选实体
7. 判断应该执行哪种动作：
   - 更新已有页面
   - 创建新页面
   - 记录新的 `view`
   - 更新人物页或主题页摘要
   - 仅在 schema 需要时更新时间线页
8. 为新内容或更新内容补充或优化 wikilinks
9. 如果创建了新页面，更新 `index.md`
10. 向 `log.md` 追加简洁记录；如果本次因不符合 domain 而丢弃资料，也应记录“discarded / out of domain”的结论
11. 在任务结尾汇报创建了哪些文件、更新了哪些文件，以及是否被判定为“未入库”

Ingest 优先级：

- 如果资料本身不属于 schema 的 domain，优先直接丢弃，不要进入 `raw/`，也不要继续做 page type 判断
- 如果资料主要帮助理解知识点，优先走 schema 中定义的“知识理解线”
- 如果资料主要帮助理解某个人的思维方式，优先走 schema 中定义的“人物学习线”
- 如果同一个人只是重复表达已有判断，优先更新已有页面，避免碎片化

Domain 判断规则：

- 先问：这份资料是否直接属于 `SCHEMA.md` 定义的领域边界
- 再问：如果不直接属于领域主题，它是否明确服务于 schema 已经允许的长期主线
- 只有上述任一答案为“是”，才继续做 page type、建页阈值和链接判断
- 如果答案为“否”，直接丢弃，不写入 `raw/`、`people/`、`concepts/`、`topics/`、`views/`、`summaries/`、`comparisons/`、`queries/`、`outputs/`

## Query 工作流

当用户提出一个应基于知识库回答的问题时：

1. 读取 `index.md`
2. 搜索相关页面
3. 阅读最相关的页面
4. 基于已整理的知识页进行综合回答
5. 如果答案具有长期价值，则在 schema 支持的情况下写入 `queries/` 或 `comparisons/`
6. 向 `log.md` 追加一条 query 记录

Query 规则：

- 优先从整理好的知识页回答
- 只有在整理层不足时，才回头重新挖掘 `raw/`
- 区分临时回答与可复用知识
- 如果某个 query 结果值得长期保留，按 `SCHEMA.md` 规定的格式写回知识库

## Lint 工作流

当用户要求审计、清理、整理或检查知识库健康状况时：

1. 检查页面连接情况
2. 检查 wikilinks
3. 检查 `index.md` 是否完整
4. 检查 frontmatter 是否完整且一致
5. 检查是否符合 schema
6. 检查是否存在陈旧页面
7. 检查是否有未解决的冲突
8. 检查页面长度，识别可拆分页面
9. 检查连接过弱的页面
10. 检查 `log.md` 是否需要轮转

Lint 输出必须包括：

- 严重问题
- 重要问题
- 清理建议
- 推荐下一步动作

## 首次运行行为

如果知识库为空，或只完成了部分初始化：

1. 若缺失则创建 `SCHEMA.md`
2. 若缺失则创建 `index.md`
3. 若缺失则创建 `log.md`
4. 若缺失则创建必需的目录结构
5. 初始化完成后，再等待第一批资料或任务

如果用户已经有偏好的 schema，不要自行发明新 schema。只创建最小脚手架，让 `SCHEMA.md` 来定义具体规则。

## 任务结束格式

每次任务结束时，必须明确说明：

1. 本次完成了什么
2. 新建了哪些文件
3. 更新了哪些文件
4. 是否仍有未解决的冲突或歧义
5. 推荐的下一步动作是什么

## 实际边界

`SKILL.md` 定义的是：

- 何时使用这个 skill
- 如何安全开始
- 如何执行 ingest / query / lint
- 如何把知识库当成长期系统来维护

`SCHEMA.md` 定义的是：

- 页面类型
- 语义路由
- frontmatter 约定
- 目录含义
- 命名规则
- 标签体系
- 建页阈值
- 链接规则
- 冲突处理细则

除非用户明确要求做成“自包含 skill 文件”，否则不要在这里重复 schema 细节。
