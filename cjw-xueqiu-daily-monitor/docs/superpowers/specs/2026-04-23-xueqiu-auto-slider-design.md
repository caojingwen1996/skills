# 雪球验证页自动滑块方案设计

## 背景

当前技能在检测到雪球验证页或登录页时，只能提示操作者在自动化 Chrome 窗口中手工处理，然后由脚本继续等待页面恢复。

这一行为主要由以下文件定义：

- `scripts/extract_xueqiu_posts.mjs`
- `references/workflow.md`
- `references/error-policy.md`

其中 `extract_xueqiu_posts.mjs` 已具备以下能力：

- 识别验证页与登录页
- 在当前自动化 Chrome 窗口中保持等待
- 页面恢复后继续当前抓取轮次

当前缺口是：遇到滑块验证时，脚本不会先尝试自动完成拖拽。

## 目标

- 在检测到雪球滑块验证页时，优先进行自动滑块尝试
- 自动验证成功后，当前账号抓取轮次继续执行，不要求人工重跑
- 自动验证失败时，保留现有人工接管路径，不破坏当前等待恢复机制
- 将该行为写入技能文档与错误策略，保证实现和文档一致

## 非目标

- 不处理登录页自动登录
- 不接入第三方打码平台
- 不承诺所有验证页都能被自动通过
- 不移除人工兜底链路

## 推荐方案

推荐采用 `CDP 鼠标事件 + DOM 定位滑块 + 自动失败回退人工` 的方案。

原因：

- 当前脚本已经通过 CDP 与自动化 Chrome 建立连接，不需要新增浏览器控制栈
- 改动集中在一个主脚本内，便于小步迭代
- 即使自动验证失败，也可以无缝回退到现有人工恢复流程

## 方案对比

### 方案 A：CDP 鼠标事件拖拽

做法：

- 检测验证页中的滑块手柄和轨道元素
- 读取元素边界框，计算拖动起点和目标终点
- 用 `Input.dispatchMouseEvent` 注入按下、移动、释放事件
- 拖动轨迹使用分段加速、减速、微抖动和短暂停顿

优点：

- 复用现有 CDP 基础设施
- 首版实现成本最低
- 便于控制失败回退

缺点：

- 成功率受页面结构和风控策略影响

### 方案 B：截图识别缺口后拖拽

做法：

- 截图验证码区域
- 识别缺口位置
- 按图像结果计算最优拖动距离

优点：

- 在 DOM 不稳定时理论上更鲁棒

缺点：

- 需要新增图像处理链路
- 首版复杂度明显更高

### 方案 C：第三方打码服务

做法：

- 将验证码交给外部识别服务

优点：

- 某些情况下通过率可能较高

缺点：

- 需要外部网络调用和凭证管理
- 与当前技能的本地化、可控性目标不一致

结论：首版采用方案 A。

## 设计细节

### 1. CLI 行为

在 `scripts/extract_xueqiu_posts.mjs` 中新增参数：

- `--verification-mode manual`
- `--verification-mode auto-then-manual`
- `--verification-mode auto-only`

默认值建议为 `auto-then-manual`。

模式定义：

- `manual`：保持现状，只做人机协同
- `auto-then-manual`：先自动尝试，失败后切回人工等待
- `auto-only`：只做自动尝试，失败即报错

### 2. 验证页识别

保留现有基于文本的验证页识别逻辑，同时新增验证页 DOM 探测函数。

DOM 探测规则：

- 允许多个候选选择器
- 允许文本兜底
- 不依赖单一 class 名

探测结果至少应包含：

- 是否存在滑块手柄
- 是否存在轨道
- 手柄和轨道的中心点与边界框
- 页面当前是否仍处于验证态

### 3. 自动拖拽

新增一组内部 helper：

- `inspectVerificationWidget()`
- `buildHumanLikeDragPath()`
- `dispatchMouseDragPath()`
- `attemptAutoVerification()`

约束：

- 拖拽前先将鼠标移动到手柄中心
- 执行 `mousePressed`
- 逐步发送 `mouseMoved`
- 末尾发送 `mouseReleased`
- 每一步都带时间间隔，不使用瞬移

拖拽轨迹要求：

- 前段加速
- 中段匀速偏快
- 末段减速
- 少量纵向抖动
- 末段包含一次小幅回拉修正

### 4. 自动失败回退

自动验证失败的判定包括：

- 未找到稳定的滑块 DOM
- 拖拽执行后页面仍停留在验证态
- 拖拽过程中页面跳转或会话异常

回退规则：

- `auto-then-manual`：记录日志后进入现有 `waitForManualActionCompletion()` 路径
- `auto-only`：直接抛错
- `manual`：不进入自动尝试

无论哪种模式，都不得提前关闭自动化 Chrome 窗口。

### 5. 文档同步

需要同步更新：

- `references/workflow.md`
- `references/error-policy.md`
- 如有必要，`SKILL.md`

文档表达应统一为：

- 登录页默认人工处理
- 验证页优先自动尝试
- 自动失败时回退到同一自动化 Chrome 窗口中的人工处理
- 页面恢复后继续当前抓取轮次

## 测试策略

优先补纯函数和轻量集成测试，不引入真实雪球环境依赖。

建议覆盖：

- `--verification-mode` 参数解析
- 验证模式默认值
- 自动验证模式下的回退决策
- 拖拽轨迹生成结果的基本性质
- 文档中对自动验证与人工兜底的描述一致性

不在首版测试中覆盖：

- 真实雪球线上验证通过率
- 真实浏览器中的像素级拖拽效果

## 风险与缓解

### 风险 1：雪球调整页面结构

缓解：

- 使用多选择器和文本兜底
- 找不到稳定元素时快速回退人工

### 风险 2：雪球识别合成轨迹

缓解：

- 采用拟人轨迹而非匀速直线
- 保留人工兜底，不阻断任务

### 风险 3：默认启用后影响现有使用者

缓解：

- 使用显式模式参数
- 默认值若需谨慎，可先保守设为 `manual`，待验证后再切到 `auto-then-manual`

## 实施顺序

1. 在 `extract_xueqiu_posts.mjs` 中加入验证模式参数和内部接口骨架
2. 实现验证页 DOM 探测与拖拽轨迹生成
3. 接入 CDP 鼠标事件拖拽
4. 将自动尝试接到现有验证处理流程前
5. 更新 `workflow.md`、`error-policy.md` 和必要文档
6. 补测试并执行最小相关检查

## 验收标准

- 遇到验证页时，脚本可根据参数决定是否先自动尝试
- 自动验证成功后，当前抓取轮次继续执行
- 自动验证失败后，在 `auto-then-manual` 模式下会回退人工等待
- 登录页行为保持不变
- 文档与实现对“自动优先、人工兜底”的描述一致
